## Commands

Package manager is **Bun** (see `bun.lock`, `bunfig.toml`).

- `bun install` — install dependencies. `bunfig.toml` enforces a 24h `minimumReleaseAge` supply-chain guard; only `@lovable.dev/vite-tanstack-config` is excluded. Confirm with the user before adding more exclusions.
- `bun run dev` — Vite dev server (SSR via TanStack Start).
- `bun run build` — production build. `build:dev` builds in development mode.
- `bun run preview` — serve the production build.
- `bun run lint` — ESLint over the repo.
- `bun run format` — Prettier write.

There is no test runner configured.

## Architecture

This is a **TanStack Start** (SSR React 19) app generated by Lovable, deployed to **Cloudflare Workers** (Nitro target). It is a calorie/macro tracker with an AI food-analysis flow and an AI coach chat.

### Build pipeline — do not duplicate plugins

`vite.config.ts` uses `@lovable.dev/vite-tanstack-config`, which already bundles `tanstackStart`, `viteReact`, `tailwindcss`, `tsConfigPaths`, `nitro`, `componentTagger`, VITE_* env injection, the `@/*` path alias, and React/TanStack dedupe. Adding any of these manually breaks the app with duplicate-plugin errors. Extend via `defineConfig({ vite: { ... } })`.

### Routing (TanStack Start file-based)

Routes live in `src/routes/`. See `src/routes/README.md` for the full convention table — `__root.tsx` is the only app shell (do not introduce `src/pages/` or `app/layout.tsx`); `routeTree.gen.ts` is auto-generated, never edit by hand. API endpoints are file routes with `server.handlers` (e.g. `src/routes/api/analyze-food.ts`, `src/routes/api/coach.ts`) — these are TanStack Start API routes, **not** Supabase Edge Functions.

### SSR error handling (two-layer)

1. `src/start.ts` registers `errorMiddleware` as a global `requestMiddleware` — catches non-`statusCode` errors and renders a fallback HTML page.
2. `src/server.ts` is the Nitro/Cloudflare entry (configured in `vite.config.ts` via `tanstackStart.server.entry`). It wraps the TanStack Start handler and additionally rewrites h3's "swallowed" 500 JSON (`{"unhandled":true,"message":"HTTPError"}`) into the same error page, because h3 silently turns in-handler throws into normal Responses that try/catch never sees.

### Server functions vs API routes

- Use `createServerFn` (see `src/lib/api/example.functions.ts`) for typed RPCs called from React components — preferred over Supabase Edge Functions per the comment in that file.
- Use file-route `server.handlers` for plain HTTP endpoints (e.g. AI gateway proxies).

### Server-only code

- Files suffixed `.server.ts` are stripped from the client bundle (`src/lib/config.server.ts`, `src/integrations/supabase/client.server.ts`).
- **Cloudflare Workers bind env at request time** — `process.env.X` read at module scope resolves to `undefined`. Always read inside a handler/function (the `getServerConfig()` wrapper exists for this reason).
- ESLint forbids `import "server-only"` (a Next.js idiom). Use the `.server.ts` suffix instead.

### Env vars

- Public (`VITE_*`) — injected via `import.meta.env`, ships to the browser. Used in `src/integrations/supabase/client.ts`.
- Private (server-only) — read from `process.env` inside handlers. Includes `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `LOVABLE_API_KEY` (used by `/api/analyze-food` and `/api/coach` to call `https://ai.gateway.lovable.dev`).

### Supabase integration (`src/integrations/supabase/`)

The directory is auto-generated by Lovable — files state "This file is automatically generated. Do not edit it directly." Three client variants for three contexts:

- `client.ts` — anon-key client for the **browser**, lazy-init via Proxy.
- `client.server.ts` — **service-role** client, bypasses RLS. Server-only, never expose to client.
- `auth-middleware.ts` — `requireSupabaseAuth` middleware for serverFns that need a per-user RLS-scoped client. Validates Bearer token via `getClaims` and injects `{ supabase, userId, claims }` into context.
- `auth-attacher.ts` — client middleware `attachSupabaseAuth` that pulls the session token from the browser client and attaches it as `Authorization: Bearer …` on serverFn calls. **Must be registered as a global `functionMiddleware` in `src/start.ts`** or the browser never sends the token.

### Client state

`src/lib/store.ts` is a custom `useSyncExternalStore`-based store backed by `localStorage` (key `ctrack-v1`). All food entries, saved meals, weights, photos, profile, and coach messages live there — there is currently no Supabase persistence wired up for app data. Mutations go through the exported helpers (`addFoodEntry`, `setTheme`, `updateProfile`, etc.), not direct `setState`.

### UI

shadcn/ui ("new-york" style, slate base, lucide icons) in `src/components/ui/`, Tailwind v4 via `@tailwindcss/vite` (already in the Lovable preset). Path alias `@/* → src/*`. Toast library is `sonner`.
